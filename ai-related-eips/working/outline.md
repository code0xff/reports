# Outline — AI-related EIPs in 2026 (with EIP-8126 deep dive)

1. **Abstract** — One paragraph: what counts as an "AI-related EIP", which ones have shipped vs. which are draft, and why EIP-8126 (AI Agent Verification) is the natural deep-dive — it sits one layer above ERC-8004 and is the spec that turns "registered agent" into "verified, risk-scored agent".
2. **Introduction** — Why "AI EIPs" became a category in 2025–2026: the rapid emergence of agentic systems on Ethereum (Coinbase, MetaMask, EF dAI Team, Google) needing identity, reputation, payments, *and verifiable trustworthiness*.
3. **The 2025–2026 AI EIP cluster** — Shipped or actively-drafted EIPs: ERC-8004 (Trustless Agents, mainnet 2026-01-29), ERC-7857 (AI Agent NFT with private metadata, 0G Labs), ERC-8183 (agentic commerce / programmable escrow), and adjacent infrastructure EIPs the agent stack depends on (EIP-8141 Frame Transaction native AA, ERC-8211 smart batching, EIP-8128 signed HTTP requests).
4. **EIP-8126 deep dive — AI Agent Verification** — Full spec: status (Draft, created 2026-01-15, by Leigh Cronian and Chris Johnson), required dependencies (EIP-155, EIP-191, EIP-712, EIP-721, EIP-3009, EIP-8004), the five verification categories (ETV, MCV, SCV, WAV, WV), the 0–100 risk-score system with five tiers, ZK-Proof-based privacy preservation, optional Quantum Cryptography Verification (QCV) using AES-256-GCM, the optional on-chain interface (`AgentVerified` event, `getLatestRiskScore` view), Sybil-resistance via ERC-8004 minting costs, security considerations, and the off-chain-first architecture.
5. **How these EIPs compose** — The agent stack as composition: identity (ERC-8004 Identity), reputation (ERC-8004 Reputation), verification + risk scoring (ERC-8126 + ERC-8004 Validation), tokenized agent assets (ERC-7857), payments (ERC-8183), authenticated off-chain I/O (EIP-8128). Where each layer plugs in.
6. **Open issues and contradictions** — Definitional drift ("AI EIP" loose), governance fragmentation, the gap between draft and production for most of the cluster, dependency on ERC-8004's mainnet stability for ERC-8126 to ship, and the "off-chain verification" trust assumption that ERC-8126 deliberately accepts.
7. **Limitations** — Vendor-stated adoption, drafts can move quickly, ERC-8126 reference implementation but no deployed addresses, our gather depended on third-party summaries for some specs (ERC-8211).
