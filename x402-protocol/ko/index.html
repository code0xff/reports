아키텍처, 결제 방식 및 확장 시스템

발행일 2026-04-28 태그 payments, protocol, blockchain, AI-agents, HTTP, stablecoins 상태 ready

HTML 다운로드

초록

x402는 오랫동안 사용되지 않던 HTTP 402 "Payment Required" 상태 코드를 활용하여 스테이블코인 결제를 표준 HTTP 요청/응답 사이클에 직접 내장하는 개방형 인터넷 네이티브 결제 프로토콜이다. Coinbase Developer Platform이 2025년 5월에 출시한 이 프로토콜은 머신 투 머신 상거래, 특히 인간의 개입이나 자격 증명 없이 컴퓨팅 자원·데이터·API 접근 권한을 자율적으로 구매해야 하는 AI 에이전트를 주요 대상으로 한다. 핵심 메커니즘은 세 가지 커스텀 HTTP 헤더(PAYMENT-REQUIRED, PAYMENT-SIGNATURE, PAYMENT-RESPONSE)와 퍼실리테이터(facilitator)라는 선택적 중개자를 활용하여 결제자의 개인 키를 노출하지 않고 온체인 결제를 검증·정산한다. 2025년 12월에 출시된 V2에서는 모듈식 SDK 아키텍처가 도입되었으며, Uniswap의 Permit2를 통해 모든 ERC-20 자산으로 지원 토큰이 확대되고, Bazaar(서비스 탐색), Sign-in-with-x(지갑 기반 세션), 가스 후원 메커니즘을 포함한 확장 시스템이 공식화되었다. 2026년 초 기준, x402 에코시스템은 복수의 EVM 체인과 Solana에 걸쳐 운영되며, 보고된 연간 결제 규모는 약 6억 달러에 달하지만 방법론과 투기적 활동 관련 중요한 주의 사항이 존재한다. 본 보고서는 공개된 문서와 코드를 기반으로 x402의 사양, 결제 방식, 확장 시스템, 멀티체인 아키텍처 및 현재 한계를 기술 참조 자료로 제공한다.

x402: HTTP 네이티브 결제 프로토콜

서론

HTTP 402 상태 코드는 1990년대 초부터 결제 요구 용도로 예약되어 있었으나, 공식 사양 없이 수십 년간 방치된 코드였다. 2024년과 2025년에 걸쳐 자율 AI 에이전트가 부상하면서 기계 판독 가능한 결제 프로토콜에 대한 새로운 수요가 생겨났다. 에이전트들은 OAuth 토큰, 청구 대시보드, 또는 인간 승인 절차 없이 API 호출·컴퓨팅 자원·데이터에 대한 비용을 지불해야 한다. 기존 결제 처리사는 최소 거래 수수료 때문에 서브달러(sub-dollar) 소액결제를 경제적으로 불가능하게 만들고, 가맹점에게 계정 개설·KYC 통과·인간 사용자를 위해 설계된 복잡한 체크아웃 흐름 통합을 요구한다.

x402는 결제를 HTTP의 일급(first-class) 기본 요소로 취급함으로써 이러한 제약을 해결한다. 클라이언트가 리소스를 요청하고 아직 결제하지 않은 경우, 서버는 기계 판독 가능한 결제 사양을 담은 표준 402 응답을 반환한다. 클라이언트는 요건을 파싱하고 서명을 생성한 뒤, 헤더에 결제 증명을 포함하여 요청을 재시도한다. 퍼실리테이터는 서명을 검증하고 정산 트랜잭션을 블록체인에 제출한다. 그런 다음 리소스 서버는 200 응답과 함께 요청된 콘텐츠를 반환한다 [1][2]. 전체 사이클은 약 2초가 소요된다 [3].

Coinbase Developer Platform은 2025년 5월 6일 x402를 오픈소스 프로젝트로 공개했다 [3][4]. 2025년 9월 Coinbase와 Cloudflare는 프로토콜을 중립적인 개방 표준으로 관리하기 위해 x402 Foundation을 공동 설립했다 [5][6]. 두 번째 메이저 버전은 2025년 12월에 출시되어 방식 및 확장 시스템을 확장하고 TypeScript SDK를 @x402 npm 조직으로 마이그레이션했다 [7][8].

핵심 프로토콜 아키텍처

구성 요소

x402 프로토콜은 네 가지 참여자 역할을 정의한다.

  • 클라이언트: 402 응답을 파싱하고 결제 페이로드에 서명할 수 있는 모든 HTTP 클라이언트 — 브라우저, AI 에이전트, CLI 도구, SDK.
  • 리소스 서버: 결제를 조건으로 콘텐츠나 API 응답을 제공하는 HTTP 서버. Express, Hono, Next.js, FastAPI, Flask, Gin 등 다양한 프레임워크용 미들웨어 라이브러리가 최소한의 설정으로 이 역할을 구현한다 [2].
  • 퍼실리테이터: /verify와 /settle REST 엔드포인트를 노출하는 중개자. 리소스 서버는 콘텐츠를 제공하기 전에 /verify를 호출하여 결제 서명을 검증하고, 이후 /settle을 호출하여 온체인 이체를 실행한다 [2][9].
  • 블록체인: 정산 레이어. 현재 지원되는 레이어로는 Base, Polygon, Arbitrum, World Chain, Solana, Stellar, Algorand가 있으며, 커뮤니티 퍼실리테이터를 통해 추가 체인도 지원된다 [2][10][11].

결제 흐름

표준적인 12단계 흐름은 다음과 같다.

1. 클라이언트가 표준 HTTP로 리소스를 요청한다. 2. 리소스 서버가 유효한 결제 헤더의 부재를 감지한다. 3. 서버는 HTTP 402를 반환하고, 수락 가능한 결제 옵션(방식, 네트워크, 토큰, 금액, 수신자, 시간 창)이 담긴 PaymentRequired 객체를 base64로 인코딩하여 PAYMENT-REQUIRED 헤더에 포함시킨다. 4. 클라이언트는 결제 옵션을 선택하고 PaymentPayload를 구성한다. 5. 클라이언트가 개인 키를 사용하여 페이로드에 서명한다(EVM의 경우 EIP-712, 다른 체인의 경우 동등한 방식). 6. 클라이언트는 base64로 인코딩된 PaymentPayload가 담긴 PAYMENT-SIGNATURE 헤더를 포함하여 요청을 재시도한다. 7. 리소스 서버가 페이로드를 퍼실리테이터의 /verify 엔드포인트로 전달한다. 8. 퍼실리테이터가 서명을 검증한다(서명자 주소 복원, 잔액 확인, 시간 창 확인, 금액 및 수신자 검증, 실행 시뮬레이션). 9. 리소스 서버가 요청을 처리한다(비즈니스 로직 실행). 10. 리소스 서버가 퍼실리테이터의 /settle 엔드포인트를 호출하여 온체인 이체를 실행한다. 11. 퍼실리테이터가 서명된 승인을 블록체인에 제출하고 확인을 기다린다. 12. 리소스 서버가 트랜잭션 해시를 포함한 정산 영수증이 담긴 PAYMENT-RESPONSE 헤더와 함께 HTTP 200을 반환한다 [1][12][9].

HTTP 헤더

모든 결제 데이터는 HTTP 헤더를 통해 전달되므로 요청 및 응답 본문은 애플리케이션 콘텐츠를 위해 자유롭게 사용할 수 있다 [7].

헤더방향내용
PAYMENT-REQUIRED서버 → 클라이언트Base64 인코딩 JSON: 수락 방식, 네트워크, 금액, 수신자, 만료
PAYMENT-SIGNATURE클라이언트 → 서버Base64 인코딩 JSON: 서명된 PaymentPayload
PAYMENT-RESPONSE서버 → 클라이언트Base64 인코딩 JSON: 정산 영수증, 트랜잭션 해시

V2 릴리스에서는 이전의 X-PAYMENT 및 X-PAYMENT-RESPONSE 접두사 헤더가 현대 HTTP 관행에 맞춘 표준화된 형식으로 교체되었다 [7].

신뢰 모델

프로토콜의 핵심 불변 조건은 클라이언트가 서명한 후 퍼실리테이터가 수신자 주소나 결제 금액을 변경할 수 없다는 것이다. 이 필드들은 클라이언트의 EIP-712 서명에 암호학적으로 결합되어 있어, 수정 시 서명이 무효화되어 검증 단계에서 거부된다. 퍼실리테이터의 역할은 사전 승인된 트랜잭션을 브로드캐스트하는 것이며, 자금을 수탁하거나 리디렉션하지 않는다 [12].

V2 아키텍처 변경 사항

2025년 12월 V2 릴리스에서는 모듈식, 플러그인 기반 SDK 설계가 도입되었다. 개발자는 SDK 코어를 수정하지 않고 독립 패키지로 새 체인, 자산, 결제 방식을 등록할 수 있다. TypeScript 패키지는 @x402 npm 조직 하에 세분화된 경계로 재구성되었다: @x402/core, @x402/evm, @x402/svm, @x402/stellar, 프레임워크 미들웨어 패키지(@x402/express, @x402/hono, @x402/next, @x402/fastify), 그리고 @x402/extensions [13][7]. Python(pip install x402)과 Go(go get github.com/x402-foundation/x402/go) SDK도 동등한 기능을 제공한다 [2].

결제 방식

x402 용어에서 방식(scheme)이란 논리적인 결제 방법론이다. 방식은 구현 측면에서는 네트워크별로 다르지만 개념적으로는 네트워크에 독립적이다. exact 방식은 "고정된 금액을 이체한다"는 의미이지만, EVM 체인에서와 Solana에서의 구현 방식은 다르다 [1]. 프로토콜은 확장 가능하며, 코어 수정 없이 새로운 방식을 등록할 수 있다.

exact 방식

exact 방식은 요청당 정확히 미리 결정된 금액을 이체한다. 현재 CDP 퍼실리테이터가 배포한 유일한 프로덕션급 방식이다. EVM 체인에서는 세 가지 자산 이체 방법을 지원한다 [12].

EIP-3009 (호환 토큰 권장): transferWithAuthorization을 구현한 토큰(주로 USDC와 EURC)은 완전히 가스 없이 이체할 수 있다. 클라이언트가 from, to, value, validAfter, validBefore, 재사용 방지 nonce를 포함한 승인 객체에 서명하면, 퍼실리테이터가 서명과 함께 token.transferWithAuthorization()을 호출한다.

Permit2 (모든 ERC-20에 대한 범용 대안): Uniswap의 Permit2 컨트랙트가 EIP-3009가 없는 토큰에 대한 범용 승인 릴레이 역할을 한다. 클라이언트가 증인 이체 승인에 서명하면 퍼실리테이터가 0x402085c248EeA27D92E8b30b2C58ed07f9E20001의 캐노니컬 프록시 컨트랙트를 호출한다 [12]. 일회성 Permit2 승인이 필요하며, 이후 결제에는 오프체인 서명만 필요하다. 가스 후원 확장(§확장 시스템 참조)으로 이 일회성 온체인 승인을 처리할 수 있다.

ERC-7710 (스마트 컨트랙트 계정): ERC-7710 위임을 구현한 계정은 위임 관리자 컨트랙트를 통해 이체를 승인할 수 있다. 검증은 시뮬레이션 기반으로, 퍼실리테이터가 redeemDelegations()를 시뮬레이션하여 유효성을 확인한다 [12].

Solana에서 exact 방식은 SPL 토큰 프로그램 이체(v1, v2)와 Token2022 프로그램 토큰을 사용한다 [10].

upto 방식

upto 방식은 클라이언트가 최대 지출 한도를 승인하고, 실제 청구 금액은 요청 완료 시점의 리소스 소비량에 따라 결정된다. 요청당 토큰 LLM 추론 청구나 대역폭 기반 API 요금 등의 종량제 가격 모델을 지원한다. CDP 퍼실리테이터는 Base, Base Sepolia, Polygon, Arbitrum, World, World Sepolia에서 upto를 지원한다 [10].

deferred 방식 _(제안됨)_

Cloudflare는 즉각적인 온체인 정산이 트랜잭션 처리량 제약으로 비실용적인 시나리오를 위한 deferred 방식을 제안했다 [5][14]. 이 모델에서는 여러 소액결제를 일정 기간 집계한 뒤 단일 주기적 온체인 트랜잭션으로 정산한다. 분당 수백만 건의 소액결제가 발생하는 웹 크롤링이나 스트리밍 데이터 접근과 같은 고빈도 사용 사례를 대상으로 한다. 2026년 4월 기준, deferred 방식은 커뮤니티 빌더와 함께 활발히 개발 중이나 주 사양에는 아직 병합되지 않았다.

확장 시스템

x402 V2는 TypeScript의 @x402/extensions와 Algorand 변형의 pip install "x402-avm[extensions]"으로 패키지화된 확장 레이어를 공식화했다. 확장은 핵심 결제 흐름과 독립적으로 리소스 서버와 클라이언트가 선택적으로 활성화할 수 있는 기능이다.

Bazaar (서비스 탐색)

Bazaar는 결제 게이팅된 API가 구조화된 기계 판독 가능 형식으로 기능을 광고할 수 있게 해주는 x402 내장 리소스 탐색 프로토콜이다 [15][16]. 리소스 서버가 bazaar_resource_server_extension을 등록하면, 런타임에 엔드포인트 메타데이터(GET/DELETE/HEAD 엔드포인트의 쿼리 파라미터, POST/PUT/PATCH의 요청 본문 스키마, 출력 스키마, 가격, 지원 네트워크)가 402 응답에 자동으로 추가된다 [16].

퍼실리테이터와 탐색 인덱스(agentic.market 포함)가 이러한 구조화된 응답을 크롤링하여 검색 가능한 결제 서비스 카탈로그를 구축한다. 서비스 운영자는 엔드포인트를 수동으로 제출할 필요 없이 Bazaar 확장과 유효한 402 응답 형식만으로 자동 인덱싱이 가능하다 [2][16].

Sign-in-with-x (SIWx) _(벤더 발표, 미출시)_

SIWx는 CAIP-122("Sign-In with Caip Account")를 기반으로 지갑 신원을 재사용 가능한 세션과 연결하는 계획된 인증 확장이다 [7][2]. 매 요청마다 완전한 온체인 결제를 요구하는 대신, SIWx를 통해 클라이언트가 지갑 제어권을 한 번 증명하고 이후 요청에 유효한 세션 자격 증명을 받을 수 있다. 이 확장은 V2 로드맵에서 "즉각적인 후속 출시" 항목으로 나열되어 있지만, 2026년 4월 기준으로 공개된 사양이나 구현이 없다.

가스 후원 확장

Permit2 기반 결제의 실질적 장벽은 새로운 결제자가 가스 없는 이체 전에 제출해야 하는 일회성 온체인 ERC-20 승인이다. x402 V2는 이 비용을 흡수하기 위한 두 가지 가스 후원 메커니즘을 도입했다 [7][2].

  • EIP-2612 가스 후원: EIP-2612의 permit() 함수를 구현한 토큰의 경우, 퍼실리테이터가 오프체인 서명된 permit 메시지를 사용하여 Permit2 승인을 온체인에서 후원하므로 구매자는 가스가 필요하지 않다.
  • ERC-20 승인 가스 후원: EIP-3009와 EIP-2612가 모두 없는 기본 ERC-20 토큰(예: USDT)의 경우, 퍼실리테이터가 소량의 ETH를 클라이언트에게 전송하여 승인 가스비를 충당한다.

결제 식별자

결제 식별자는 프로토콜 수준에서 멱등성(idempotency) 보장을 제공한다. 클라이언트가 안정적인 식별자를 PaymentPayload에 포함하면, 리소스 서버가 중복 제출(예: 네트워크 타임아웃 후 재시도된 요청)을 이중 청구 없이 감지하고 거부할 수 있다 [15].

A2A x402 확장 (에이전트 간 결제)

Google의 Agentic Commerce 팀은 x402를 에이전트 간(A2A) 통신 프로토콜로 확장하는 사양과 Python 구현을 공개했다 [17]. A2A x402 확장은 세 가지 메시지 흐름을 정의한다.

1. payment-required: 판매자 에이전트가 결제 조건을 구조화된 A2A 메시지로 전송한다. 2. payment-submitted: 클라이언트 에이전트가 서명하여 결제 증명을 반환한다. 3. payment-completed: 판매자가 온체인에서 결제를 검증하고 서비스를 제공한다.

이 라이브러리는 에이전트 프레임워크에 대한 흐름 자동화를 위한 실행자 미들웨어를 갖춘 "함수형 코어, 명령형 셸" 아키텍처를 따른다. 사양 수준에서는 언어 독립적이다 _(단일 출처 — 미검증)_ [17].

MCP (Model Context Protocol) 통합

x402는 Anthropic의 Model Context Protocol과 통합하여 AI 에이전트 프레임워크에서 도구별 수익화를 가능하게 한다 [5][18]. MCP 서버는 개별 도구 수준에서 결제 요건을 지정할 수 있으며, 일부 도구는 무료이고 일부는 유료인 "혼합 모드"도 지원한다. 클라이언트 측 x402 전송은 402 감지를 자동으로 처리하여 구성된 서명자에게 결제 요건을 제시하고 사용자 개입 없이 재시도한다 [18].

Algorand/AVM 확장

커뮤니티 구현체(x402-avm, PyPI 패키지)는 Algorand 제네시스 해시 형식(예: 테스트넷의 경우 algorand:SGO1GKSzyE7IEPItTxCByw9x8FmnrCDexi9/cOUJOiI=)을 사용한 네트워크 식별과 함께 ExactAvmServerScheme을 추가하여 x402를 Algorand Virtual Machine으로 확장한다 [16]. 이 패키지는 Bazaar 확장과 FastAPI/Flask 프레임워크 미들웨어를 번들로 포함한다.

멀티체인 및 토큰 지원

네트워크 식별

x402는 모든 네트워크 참조에 CAIP-2(Chain Agnostic Improvement Proposal 2) 식별자를 사용하여, 단일 클라이언트 구현으로 일관된 문자열 형식을 통해 여러 블록체인을 대상으로 할 수 있다. 예시: eip155:8453(Base 메인넷), eip155:137(Polygon), solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp(Solana 메인넷) [2][10].

CDP 퍼실리테이터 지원 현황

2026년 4월 기준 Coinbase Developer Platform 퍼실리테이터(주요 프로덕션 퍼실리테이터)는 다음 네트워크를 지원한다 [10].

토큰 지원

EVM 네트워크에서 x402 V2는 Permit2를 통해 모든 ERC-20 토큰을 지원하며, EIP-3009 호환 토큰(USDC 및 EURC)은 완전한 가스 없는 처리가 가능하다. 이는 EIP-3009 희소성으로 인해 사실상 USDC로만 제한되었던 v1 범위를 확대한 것이다 [19][10]. Solana에서는 SPL 토큰 프로그램(v1, v2)과 Token2022 토큰이 지원된다 [10]. Stellar 통합은 USDC, PYUSD, USDY를 브릿지되거나 랩핑된 자산이 아닌 네이티브 자산으로 지원한다 [11].

커뮤니티 확장 네트워크

에코시스템 페이지는 Stellar, Algorand, TRON, BNB Chain, Bitcoin을 포함한 추가 네트워크로 x402를 확장하는 커뮤니티 퍼실리테이터를 나열한다 [20][11][16]. 프로토콜 사양은 이러한 기여를 명시적으로 환영하며, 보안상의 이유 없이는 기존 네트워크 지원을 제거하지 않는 하위 호환성 보장을 정의한다 [1].

에코시스템, 거버넌스, 도입 현황

x402 Foundation

2025년 9월 Coinbase와 Cloudflare는 프로토콜의 독립적 관리 기구인 x402 Foundation을 공동 설립했다 [5][6]. Foundation의 사명은 도입을 촉진하고 프로토콜의 중립성을 유지하는 것이다. 2026년 4월 기준, Foundation의 구조 세부 사항(법적 형태, 거버넌스 규칙, 멤버십)은 공식적으로 발표되지 않았다 _(벤더 발표)_ [5].

에코시스템 파트너

x402 에코시스템 페이지는 AWS, Cloudflare, Stripe, Vercel, World, Google, Visa, Circle을 x402에 대한 지원을 발표하거나 통합한 조직으로 나열한다 [6][5]. Cloudflare가 가장 문서화된 기술적 통합을 보유한다. 다른 파트너들의 통합 깊이는 다양하다 [20][6].

SDK 및 툴링

프로덕션 수준의 SDK는 TypeScript(@x402 npm 조직의 모듈식 패키지), Python(pip install x402), Go로 제공된다. Rust(axum 지원), Java 등의 커뮤니티 구현도 존재한다 [2][13][20]. 에코시스템에는 50개 이상의 퍼실리테이터, 대시보드(x402station), 프록시 CLI(x402-proxy), 서비스 탐색 도구(x402search, EntRoute, 13,000개 이상의 인덱싱된 API를 보유한 OpenClaw)가 포함된다 [20].

도입 지표

보고된 도입 수치는 출처와 시간대에 따라 크게 다르며, 실제 성장과 방법론적 차이를 모두 반영한다.

  • CryptoSlate (2025년 12월): 7개월간 총 7,500만 건의 트랜잭션, $2,400만 정산 [8].
  • Sherlock (2026년 초): Base에서 1억 1,900만 건 이상의 트랜잭션, 연간 6억 달러의 결제 규모 [3].
  • Solana Foundation: 2025년 여름 출시 이후 Solana에서 3,500만 건 이상의 트랜잭션, 1,000만 달러 이상의 규모 [21].
  • MEXC News: 연간 6억 달러의 집계 규모 [22].

2025년 12월과 2026년 2월 사이에 보고된 일일 트랜잭션 규모의 92% 급락(평균 73만 1,000건에서 5만 7,000건으로)은 초기 피크 수치가 진정한 프로토콜 사용보다는 투기적 활동이나 테스트 활동을 반영했을 가능성을 제기한다 ^s16.

한계 및 논의

결제 비가역성

x402 결제는 온체인 정산 후 돌이킬 수 없는 푸시 결제이다. 프로토콜에는 네이티브 환불 메커니즘이 정의되어 있지 않다. CDP FAQ는 환불을 위해서는 별도의 외부 이체나 미래의 에스크로 기반 조건부 결제 확장이 필요하다고 명시적으로 인정한다 [15].

퍼실리테이터 신뢰 요구 사항

퍼실리테이터가 결제 금액이나 수신자를 변경할 수 없음에도 불구하고(암호학적으로 결합되어 있음), 클라이언트는 여전히 퍼실리테이터가 (a) 정산 결과를 정확하게 보고하고, (b) 정산을 지연하거나 억제하지 않으며, (c) 가용성을 유지할 것이라고 신뢰해야 한다. 프로토콜의 신뢰 최소화 설계가 퍼실리테이터를 운영 의존성의 단일 실패 지점으로서 제거하지는 않는다 [1][12].

거버넌스 투명성

coinbase/x402 저장소의 공식 ROADMAP.md 파일에는 "(update coming soon)"이라는 플레이스홀더 텍스트만 포함되어 있다 _(단일 출처 — 미검증)_ [23]. x402 Foundation의 구조 세부 사항은 2026년 4월 기준으로 여전히 미공개 상태이다 [5].

지표 신뢰성

트랜잭션 규모 지표는 상당한 불확실성을 내포한다. 공개된 피크 수치에는 투기적 활동이 포함된 것으로 보인다: CryptoSlate는 프로토콜에 네이티브 토큰이 없음에도 불구하고 "x402"라는 브랜드를 단 투기 토큰이 거래소에 상장된다고 지적하며 [8], 2026년 초의 92% 규모 급락은 초기 피크 수치가 토큰 투기를 반영했음을 시사한다 [14].

확장 미성숙도

v1에서는 EIP-3009의 희소성으로 인해 사실상 USDC만 사용 가능했으며, Permit2를 통한 모든 ERC-20으로의 확대는 V2의 기능으로 원활한 사용을 위해 가스 후원 확장이 필요하다 [19]. 실제 도입에 중요한 Sign-in-with-x와 deferred 결제 방식은 2026년 4월 기준 아직 출시되지 않았다.

보안 감사 현황

리서치 과정에서 x402 프로토콜 사양이나 CDP 퍼실리테이터 구현에 대한 공식 제3자 보안 감사의 공개 기록을 찾지 못했다 [14]. 퍼실리테이터가 실제 스테이블코인 결제의 온체인 정산을 처리한다는 점에서, 이는 잠재적 통합자들에게 중요한 위험 고지 사항이다.

경쟁 접근 방식

L402 (Lightning Network): Lightning Labs의 L402 사양은 Lightning Network 인보이스와 마카룬 토큰을 접근 자격 증명으로 사용하여 동일한 HTTP 402 코드를 재활용했다. L402는 수년간의 프로덕션 성숙도와 검증된 규모를 갖고 있지만 Bitcoin 에코시스템에 한정되어 체인 독립적이지 않다 [14].

EVMAuth: ERC-1155 토큰을 시간 제한적, 역할 기반 접근 자격 증명으로 사용하는 새로운 표준이다. EVMAuth는 x402의 결제 중심 모델보다 더 세분화된 권한 제어(구독 등급, 위임, 양도 가능한 접근)를 제공한다. 두 프로토콜은 서로 다른 문제 영역을 다루므로 상호 보완적일 수 있다 [14].

Stripe / Visa 에이전트 결제: Stripe와 Visa 모두 스테이블코인 지원 및 에이전트 결제 툴링을 구축하고 있다. 이러한 접근 방식은 기존 금융 인프라의 신뢰 보장을 갖추고 있지만, 계정 요구 사항·KYC·최소 거래 금액 등의 마찰도 수반된다 [14].
