Virtual Protocol: technical foundations

Anatomy of an agent protocol across $VIRTUAL, GAME, ICV, and ACP

Abstract

Virtual Protocol is a web3 agent infrastructure that tokenizes autonomous AI agents as on-chain assets and reshapes them into units that can be jointly owned and traded. Its technical stack resolves into four layers: (1) per-agent ERC-20 tokens with a fixed supply of one billion per agent, paired with $VIRTUAL; (2) a lifecycle that moves from bonding curve to a 42,000-$VIRTUAL graduation event into a Uniswap V2 pool with a ten-year LP lock; (3) the Immutable Contribution Vault, which anchors each agent's cognitive, voice, and visual cores together with every individual contribution (Service NFTs) in an on-chain vault; and (4) the Agent Commerce Protocol, which handles agent-to-agent commerce with a four-stage state machine (Request / Negotiation / Transaction / Evaluation) and smart-contract escrow. The cognitive loop itself runs off-chain on the open-source Python/Node.js GAME SDK, while ownership, payment, and auditability remain on-chain. The protocol launched in October 2024 with Base as its main deployment chain and has since expanded to Solana via the Stargate bridge and Meteora pools, with agents also deployed on Ethereum, Ronin, and Arbitrum. PeckShield has conducted two audits, but direct on-chain verification of the smart contracts, post-audit security review, and long-term sustainability assessment of the agent economy remain follow-up tasks outside this report's scope.

Virtual Protocol: technical foundations

Introduction

Virtual Protocol (officially Virtuals Protocol) is a web3 agent infrastructure designed to tokenize autonomous AI agents as on-chain assets and to bundle the service exchanges between them into a single economic system. The opening line of the official whitepaper defines the protocol as "a coordinated on-chain ecosystem where autonomous agents create services or goods and participate in commerce with humans and other agents"[1]. Agents are not black-box SaaS products operated by a single team; they are restructured into units that can be jointly owned and traded via dedicated ERC-20 tokens[2][3].

The main deployment chain is Base, the Ethereum L2 operated by Coinbase, on which the protocol launched in October 2024[3][4]. Since then a multichain strategy has expanded the protocol starting with Solana, and as of 2026 more than 18,000 agents are reportedly deployed across Base, Ethereum, Solana, Ronin, Arbitrum, and other chains[5]. The Solana expansion is not a migration of a single token treasury but a bridge-based model: $VIRTUAL moves between chains via the Stargate bridge and is traded on Solana through Meteora's liquidity pools[6][7].

This report focuses on Virtual Protocol's technical stack — the on-chain token layer, the GAME agent runtime, the Immutable Contribution Vault (ICV), the Agent Commerce Protocol (ACP), and auditability. Market performance, price, competitive comparisons, and the economic design of agent staking and retirement are deferred to follow-up research.

Core Architecture

Virtual Protocol's architecture divides into four layers: (1) the base token $VIRTUAL, (2) per-agent ERC-20 tokens with paired liquidity, (3) the Immutable Contribution Vault to which agent components are bound, and (4) the GAME framework that runs the agent's decision loop. The first three live on-chain; the fourth is primarily off-chain.

Pairing $VIRTUAL with the agent token. Whenever a new agent is created, one billion units (fixed supply of 1,000,000,000) of an ERC-20 token unique to that agent are minted[8][2][4]. The agent token has no independent market price; it is traded only as a pair with $VIRTUAL, so demand for the agent token translates directly into demand for $VIRTUAL. After an agent graduates, its Uniswap V2 LP is staking-locked for ten years, so initial liquidity is structurally bound to the agent[9].

The GAME framework. GAME (Generative Autonomous Multimodal Entities) is a modular agent framework designed so that an agent can plan and decide autonomously on the basis of goals, personality, context, and available actions[10]. The official documentation describes a hierarchical architecture composed of a "high-level planner" and a "low-level planner"; the framework itself is exposed to external developers as an open-source SDK ("GAME SDK") and a low-code hosting service ("GAME Cloud")[11]. The framework supports an "on-chain transactions" plugin at the chain-abstraction level, but the decision loop itself is designed as a conventional LLM-driven inference system running off-chain[11].

The Immutable Contribution Vault (ICV). An agent's actual "brain, voice, and visuals" are anchored in an on-chain structure called the ICV. According to the official documentation, the ICV is organised in layers: "smart contract wallet ownership as the base layer, individual VIRTUAL agents as ERC-6551 NFTs, cognitive, voice, and visual cores stored beneath each agent, and Service NFTs of each core where approved contributions are stored"[12]. Models, data, and IP submitted by each contributor are minted as Service NFTs, bound beneath the corresponding core, and that binding is registered in the smart contract and becomes the basis for revenue distribution[12]. The Agent Creation guide, however, describes the same structure as an "ERC-1155 token-bound address", so the canonical notation across documents is not fully consistent[8]. Without performing an on-chain contract verification — which is outside this report's scope — we record both notations and do not declare which one reflects the final implementation.

On-chain / off-chain boundary. The current architecture keeps ownership (ICV's ERC-6551 / ERC-1155 NFTs), payment and settlement (ACP smart-contract escrow[13]), and liquidity and governance (the Uniswap V2 LP plus the agent token) on-chain, while delegating runtime computation such as inference, voice, and memory to the off-chain GAME runtime and its plugins[11]. The official documentation does not summarise this boundary in a single sentence; the statement above is therefore an interpretation synthesised across several pages _(unverified — no single consolidated source)_.

Agent Lifecycle

An agent's starting point is the bonding curve. Paying 100 $VIRTUAL (non-refundable) mints the new agent's ERC-20 token and opens trading on the bonding curve[8]. When the bonding curve has accumulated 42,000 $VIRTUAL, the agent "graduates": a Uniswap V2 pool pairing the agent token with $VIRTUAL is created automatically, and its LP tokens are locked for ten years while earning emission rewards through staking[9][8]. The 42k threshold and the ten-year lock are protocol-level invariants shared by every agent launch[9].

Layered on top is the distribution mechanism, Genesis Launch. Genesis is a proof-of-contribution distribution that prioritises launch allocations for community participants who have accumulated "Virgen" points: each user may commit up to 566 $VIRTUAL, a total commitment of 21,000 $VIRTUAL is targeted, and if that threshold is not met the agent is not launched and both the $VIRTUAL and the Virgen points are refunded[14]. Return of principal on failure is the core safety mechanism of Genesis[14]. The launchpad module itself has been reorganised several times between 2024 and 2025 (Genesis → Unicorn and so on) and is documented today as a modular system that developers configure to fit each project.

The concrete supply policy for agent tokens (lock/vesting, retirement handling) and the parameters of contributor staking and revenue share lie outside the technical-stack scope of this report and are left for follow-up research.

Interoperability & Runtime

Agent Commerce Protocol (ACP). ACP is the agent-to-agent commerce protocol that lets multiple agents buy and sell each other's services. The official whitepaper states that ACP is designed as "a smart contract-based escrow system, cryptographic verification of agreements, and an independent evaluation phase"[15]. The technical deep-dive divides a single transaction into a four-stage state machine: (1) Request — the agent signals intent; (2) Negotiation — terms are cryptographically signed to produce a Proof of Agreement (PoA); (3) Transaction — payment and deliverable are escrowed simultaneously in a smart contract; (4) Evaluation — the outcome is judged against the agreed criteria and reputations are updated[13]. The design also introduces an optional third-party "evaluator" agent role that can adjudicate whether the output is acceptable[13].

By design, both payment and proof of contract fulfilment are recorded on-chain in ACP, so even though the GAME runtime reasons off-chain, "who promised what at what price" and "how settlement was carried out" remain reconstructable from the chain[13]. ACP v2, announced in late 2025, introduces Jobs schemas that can be defined flexibly per domain, which the official documentation presents as relaxing v1's single-global-schema limitation.

Multichain expansion. Virtual Protocol started on a single chain (Base) and moved to a multichain architecture with the official Solana expansion in early 2025. The $VIRTUAL token itself can be moved between Base and Solana through the LayerZero-family Stargate bridge, and on Solana Meteora's DLMM pools have been adopted as the official initial liquidity layer[6][7]. As of 2026, agents are reportedly deployed on Ethereum mainnet, Ronin, and Arbitrum as well[5]. At the documentation level, however, we could not locate a public page consolidating the source of truth for $VIRTUAL supply and minting authority across each chain — the details of this integrated model require follow-up on-chain verification.

GAME runtime and developer access. The GAME SDK is open-sourced for both Python and Node.js. The official GitHub organisation (Virtual-Protocol) and the SDK-specific organisation (game-by-virtuals) host game-python and game-node repositories[16][17][18], and the whitepaper itself states that the SDK is "an open-sourced repo that allows developers full customizability"[11]. This degree of openness supports the claim that Virtual Protocol agents are not merely a hosted SaaS but retain the character of a "framework" within which external developers can build their own runtimes.

Limitations

Virtual Protocol received two PeckShield audits, dated 10 March 2024 and 31 October 2024, and both audit reports are available as downloadable PDFs from the whitepaper resources page[19][20][21]. However, (a) as of this report's research date (2026-04-18) we could not locate any publicly disclosed additional audits (for example by CertiK), and (b) tracing the severity distribution, open issues, and remediation commits inside the audit reports is outside this report's scope.

Other limitations:

  • This report draws only on the official whitepaper text and public news and technical articles. Deployment addresses and bytecode/source verification of contracts such as AgentFactory, Bonding, and the $VIRTUAL token on Basescan/Etherscan/Solscan were not performed.
  • Inconsistent standard notation was observed in the official documentation. The ICV page (s08) describes agent NFTs as "ERC-6551 NFTs", while the Agent Creation page (s14) describes them as "ERC-1155 token-bound addresses". The two descriptions may be compatible, but confirming the final implementation would require on-chain contract-level verification.
  • Staking, rewards, and retirement of agent tokens, and the long-term liquidity sustainability of the bonding-curve economy (for example, price discovery for low-cap agents), were explicitly excluded from scope A. Paid research from Messari, DWF Labs, and similar outlets could not be fully quoted due to access restrictions.
  • No security-model analysis of whether the ACP evaluator role is genuinely independent (collusion, reputation laundering, and so on) was performed; we cite only the fact that the official documentation specifies introduction of the evaluator role.
  • Because the sector's document and infrastructure cycle is fast, the technical statements in this report are valid only as of 2026-04-18.
